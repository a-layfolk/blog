# 数据链路层

# 数据链路层的设计问题

数据链路层要完成一些功能：

  • 向网络层提供一个定义良好的服务接口
  • 处理传输错误
  • 调节数据流，确保慢速的接收方不会被快速的发送方淹没

# 提供给网络层的服务

数据链路层通常会提供以下 3 种可能的服务：

  • 无确认的无连接服务

    适合两种场合：

    • 错误率很低
    • 实时通信、比如语音传输

  • 有确认的无连接服务

    适合用不确定的信道，比如无线系统 WIFI

  • 有确认的有连接服务

    保证每个帧只被接收一次，并且所有的帧都将按正确的顺序接收。

    适用于长距离且不可靠的链路，比如卫星信道或者长途电话电路。

# 成帧

数据在网络上是以很小的称为帧（Frame）的单位传输的。

数据链路层会把比特流拆分为多个离散的帧，并为每个帧提供一个称为校验和的短令牌，将校验和放在帧种一起传输。

当帧到达目标机器时，会重新计算校验和，如果新算出来的校验和与原校验和不同，则说明传输过程中出现了错误。

拆分比特流

拆分比特流有四种方法

  • 字节计数法

    利用头部的一个字段来标识该帧中的字符数

    缺陷：

    • 可能因为一个传输错误而被弄混

  • 字节填充的标志字节法

    每个帧用一个特殊字节作为开始和结束。特殊字节称为标志字节。

    标志字节如果出现在数据中，需要在标志字节前面添加转义字节（ESC），接收时需要删除转义字节，称为字节填充。

    如果转义字节出现在数据中，那么同样要在转义字节前面增加转义字节。

    类似的协议有 PPP 协议。

    缺点：

    • 一帧的长度要取决于所携带的数据内容
    • 有转义字节则需要长度会增加

  • 比特填充的标志比特法

    每个帧的开始和结束由一个特殊的比特模式，0111 1110 或十六进制的 0x7E 标记。这种模式是一个标志字节。

    每当发送方数据的数据链路层，在数据中发现连续 5 个 1，它们便自动在输出的字节流中填入一个比特 0。

    接收方看到 5 个连续的比特 1，后面紧跟着一个 0 ，它就自动剔除 0

    缺点：

    • 一帧的长度要取决于所携带的数据内容
    • 有转义字节则需要长度会增加

  • 物理层编码违禁法

    比特编码成信号通常包括一些冗余比特。我们可以利用这些保留的信号来只是帧的开始和结束

# 差错控制

如何确认所有的帧最终都传递给目标机器的网络层，并且保持正确的顺序？

  • 常用方法是向发送方提供一些反馈信息

如果数据帧完全丢失了，发送方一直发送怎么办？

  • 在数据链路层引入计时器，超时未接到反馈，则说明帧已经丢失

帧丢失后要重发，如果有的帧被发送了多次怎么办？

  • 一般有必要给发送出去的帧分配序号，这样接收方可以根据帧的序号有效区分原始帧和重传帧

    # 流量控制

如果发送方发送帧的速度超过了接收方能够接受这些帧的速度，发送方应该如何处理？

  • 第一种方法是，基于反馈的流量控制，（feedback-based flow control）接收方给发送方返回信息，允许它发送更多的数据，或者至少高速发送方自己情况怎么样。
  • 第二种方法是，基于速率的流量控制，（rate-based flow control），使用这种方法的协议有一种内置机制，它能限制发送方传输数据的速率，无需接收方的反馈信息。

# 差错检测和纠正

传输错误非常普遍，我们应该如何处理传输错误呢？

  • 纠错码（error-correcting code），也常称为前向纠错（FEC，Forward Error Correction）
  • 检错码（error-detecting code）

无论是纠错码还是检错码都无法处理所有可能的传输错误，因为提供保护措施的冗余可能像数据比特一样出现错误。（可危及它们的保护作用）。

记住两点：

  • 纠错和检错的问题出现在任何层
  • 差错编码是应用的数学，除非特别熟悉伽罗瓦领域或稀疏矩阵的性质，否则不要自己设计编码

# 纠错码

块码、码字（n 位） = 校验位（r 位）+ 数据位（m 位）

码率：不包含冗余部分所占的比例—— n/m

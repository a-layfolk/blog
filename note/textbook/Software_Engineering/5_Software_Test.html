# 软件测试

软件测试是软件开发的最后一个流程。

# 软件测试基础

软件测试是在一定软件环境下，以最小的成本来验证系统能否按照需求正确运行，并尽可能发现存在的错误。测试只能证明程序有错，而不能保证程序无措。

# 软件测试过程模型

软件测试过程主要包括：

  • 测试对象。通常为代码，现代软件工程中也包括文档。
  • 如何测试。要求有测试配置，测试环境配置，执行测试过程还有测试结果与预期结果比对。

为了协调软件开发和软件测试，需要考虑软件测试过程模型。

测试的 V 模型：

V 模型的重要价值在于，它定义了软件测试如何与软件工程各阶段相融合。

V 模型中，软件开发和软件测试是并行的。

测试的 W 模型：

W 模型是 V 模型的改进。它明确了软件开发各阶段都需要测试，而不仅仅是在编码结束后才开始。而且，测试对象不仅是代码，还可以是文档。

测试的 H 模型：

H 模型是对 W 模型在更高层次上的线性抽象。它明确了，在任何一个开发阶段，只要有必要且测试配置已准备就绪，就可以进行测试。

W 模型和 H 模型对 V 模型的演化，目的都是强调软件测试的尽早性。

# 软件测试分类

软件测试分为静态测试和动态测试两类。

就源程序而言，静态测试不需要运行就可以找出错误，动态测试则需要运行才能找出错误。

  • 静态测试
    • 桌面检查
    • 代码检查，参照文档检查代码，目的是检查错误而不是如何修改。
    • 代码走查，将测试用例输入电脑，人工查看，一般是确定程序逻辑是否正确。
  • 动态测试
    • 白盒测试。针对程序内部逻辑结构进行测试。关心程序内部的细节，
    • **黑盒测试。**将模块作为一个整体进行测试，只关注模块的输入输出。

# 白盒测试

白盒测试是基于文档的测试。它的测试用例设计是关键。其设计原则是：保证所有判断分支、循环体、边界值和独立路径都至少执行一次。

白盒测试技术又分为：逻辑覆盖、循环测试和路径测试。

# 逻辑覆盖

  • 语句覆盖，程序中每条语句至少执行一次
  • 判定覆盖，所有判定的分支至少执行一次
  • 条件覆盖，判定中的关系式的值，取 真/假 至少一次
  • 判定/条件覆盖，所有判定分支至少执行一次，判定中关系式的值也取 真/假 至少一次
  • 条件组合覆盖，比如有判定式的关系式为 C1 && C2，C3 && C4，则要把它们的 C1C2 的 4 种真假组合，C3C4 的 4 种真假组合全部执行
  • 点覆盖，在逻辑覆盖中指的就是语句覆盖
  • 边覆盖，在逻辑覆盖中指的就是判定覆盖

# 循环测试

结构化的程序设计中有三种循环：简单循环、顺序循环和嵌套循环。对这三种循环有不同的测试策略，可以看书 151 页。

# 路径测试

路径测试分为三步:

  • 计算程序流图的环形复杂度。
  • 确定需要执行进行测试的程序独立路径。
  • 根据测试路径设计测试用例。
# 环形复杂度

首先要把程序转换为程序流图。

环形复杂度 V(G) 有三种计算方法：

  • V(G) = E - N + 2，即 边的数量 - 点的数量 + 2
  • V(G) = P + 1，即判定节点的数量 + 1
  • V(G) = 流图中封闭区域的个数。可以人为添加一条从结束节点到开始节点的“虚”控制流
# 程序独立路径

流图的环形复杂度定义了程序独立路径的数量。

TIP

程序独立路径是指从程序入口到出口多次执行中，每次至少有一个语句是新的。即每次都要走过一条从未经过的边。

# 黑盒测试

完成黑盒测试的成员组成最好不要涉及程序员。

以下是几类黑盒测试技术：

  • 等价类划分。有两种情形，有效等价类和无效等价类。
    • 有效等价类是指对程序而言，有意义且合理的数据。
    • 无效等价类是指对程序而言，无意义的数据。
  • 边界值分析。
  • 错误推测法。不同于其他方法，错误推测法无规律可循，全凭经验和直觉。
  • 因果图法。列出因果，画出因果图，将因果图转换为判定表，再给出判定表中的每组的测试用例。

WARNING

白盒测试又称结构测试、逻辑驱动测试或基于程序的测试。

黑盒测试又称为功能测试或行为测试。

原本这种别称我是不想记的，但是 2018 年的软工考试试卷上出现了使用别称作为迷惑的选择题。

# 白盒测试和黑盒测试比较

技术 人员 文档
白盒 关心局部 程序员和测试人员 需要详细的设计文档
黑盒 关心整体 较多的人，比如用户和非技术人员 需要需求规格说明和概要设计说明
优点 缺点
白盒 发现大量错误，有较多工具，部分用例能自动生成 相当数据要人工生成，理解代码工作量大。仅适用单元测试
黑盒 不需要了解细节，容易生成用例，使用任何阶段 容易重复、遗漏某些错误

# 软件测试策略

软件测试从局部到全局，包括：

  • 单元测试

  • 集成测试

  • 确认测试

  • 系统测试

    系统测试是几乎涵盖了前述测试的所有内容，主要包括：

    • 功能测试
    • 性能测试
    • 压力测试
    • 容量测试
    • 安全测试
    • 文档测试
    • 恢复性测试
    • 备份测试

    这里还有一个 α 测试和 β 测试的概念：α 测试是指开发环境下，用户和软件开发人员、测试人员共同对系统进行的测试。β 测试是在系统实际用户使用环境下的进行的测试，整个过程由用户独立完成。

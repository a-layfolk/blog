# 软件危机与软件过程

为了克服软件危机， 才提出了软件工程。

软件危机是指：

  • 软件开发成本急剧增加
  • 软件开发周期难以确定
  • 软件开发过程难以控制
  • 软件质量难以保证
  • 软件维护难以为继

软件危机产生的原因体现在两方面：

  • 一方面是软件自身的特点
  • 另一方面是开发软件和使用软件的人员
    • 对软件开发缺乏正确的理论指导
    • 软件人员与用户缺乏充分交流
    • 对软件开发过程缺乏整体认识
    • 对软件产品缺乏有效一致的质量评价标准

软件工程的发展目前为止经过了四个阶段：

  • 传统软件工程阶段： 从 “手工作坊式” 到用工程化的思想指导
  • 面向对象软件工程阶段： 面向对象的思想融入到软件开发的各个过程
  • 过程工程的软件工程阶段： 开始对软件过程进行管理（开发过程， 维护过程等）
  • 构建工程的软件工程阶段： 软件重用和软件构建技术逐步成为软件工程的主要研究内容

# 软件过程模型

软件过程模型是一种软件开发策略， 这种策略对软件生命周期的各规范提出相应的过程规范， 使软件开发进度按照项目管理的预期进行。

# 软件过程模型的比较

软件生命周期模型 说明 优势 不足
瀑布模型 线性模型， 和软件生命周期各阶段对应 总览全局， 明确需求， 文档规范， 过程严谨 难以适应需求的变更
原型模型 先做出一个简单系统用以确定用户需求， 是一个压缩循环的瀑布模型。 原型模型不适合开发大型软件项目。 通过原型开发和讨论， 逐步明确问题定义和系统需求 原型的开发会增加系统成本
增量模型 将需求分解为若干部分， 每部分按照瀑布模型开发。 核心功能明确， 灵活，降低风险，可拓展 需要各类人员参与， 增加管理难度
螺旋模型 是迭代式开发过程， 将原型模型和扑瀑布模型结合， 引入风险分析机制。 螺旋模型适合大型软件项目开发。 增加风险分析，最大限度减少系统开发风险 风险分析策略需要分析人员具有经验，增加管理难度
喷泉模型 软件过程的每个阶段相互重叠， 没有明显界限。喷泉模型主要用于面向对象软件开发 各阶段平滑过渡 各阶段并行， 导致需要较多的软件人员参与到开发， 管理困难，且增加了文档审核和管理困难
敏捷过程模型 综合瀑布模型和原型模型的优点， 保证减少错误的前提下， 快速得到用户系统。 快速开发， 及时响应用户反馈 缺乏细致准备， 会造成后续维护变更困难
渐进交付迭代模型 面对不确定的需求（原型模型）， 非整体的开发（增量模型）， 影响软件质量的风险（螺旋模型），使用 MVP (Minimum Viable Product, 最小可行产品) 方法。 软件不断演进地循环往复，开发最小功能集合， 有最小开发成本。 大型复杂系统如操作系统难以做到开发最小可用系统后不再拓展，不适合对需求明确的系统采用
微软解决框架过程模型 MSF 模型为大型项目准备， 有 4 阶段， 体现瀑布模型的过程， 每一次各阶段的迭代都是螺旋模型式的提升 同步和稳定策略推动项目、技术的成功 较少在微软之外的公司得到推广

# 结构化开发方法

包括 SA, SD, SP 三个阶段：

  • SA (Structure Analysis) 结构化分析
  • SD (Structure Design) 结构化设计
  • SP (Structure Programming) 结构化编程

# 面向对象开发方法

面向对象公式的定义：

面向对象 = 对象 + 类 + 继承 + 消息

面向对象的软件工程（OOSE）就是以对象为基础， 集合上述公式的面向对象特性， 提出了 OOA, OOD, OOP 等开发过程：

  • OOA (Object-Oriented Analysis) 面向对象分析、
  • OOD (Object-Oriented Design) 面向对象设计
  • OOP (Object-Oriented Programming) 面向对象编程

这些阶段都用到 面向对象的统一建模语言（UML） 和基于 UML 的面向对象开发方法。

TIP

需要注意的是， 《软件工程基础》（胡思康， 第三版）这本书后续分别介绍的就是结构化开发方法和面向对象开发方法。

我个人在实际开发中常用到的是面向对象开发方法， 结构化开发方法好像没咋用过。

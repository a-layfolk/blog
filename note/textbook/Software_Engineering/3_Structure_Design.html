# 结构化设计方法

结构化设计 （Structure Design, SD）的基础是模块。

结构化设计的基本思想是：基于模块独立性和信息隐藏原则， 自顶向下， 逐步求精， 分解与抽象相结合， 并应用结构化程序设计技术而进行的软件设计。

结构化设计一般分为两个阶段：

  • 第一阶段完成对软件结构设计， 从而得到系统软件结构图；

  • 第二阶段得到软件系统的详细设计

# 面向数据流的设计方法

结构化设计以数据流图为基础， 根据对数据流图理解的不同， 以及对数据流图自动边界划分的不同， 分为变换分析法、事务分析法和混合分析法。

# 变换分析法

变换分析法是一个迭代的分析过程。 具体内容看课本 101 页。

变换分析法分为六个步骤：

  1. 复审并精细化数据流图

  2. 划分自动化边界， 确定数据流特征， 判断数据流是变换流还是事务流。

    TIP

    变换流的特征是数据有明显输入、 处理和输出过程， 在处理部分没有过多的控制和判断

  3. 划分数据输入输出边界， 分离出处理部分

    TIP

    输入、 输出边界不同于自动化边界， 自动化边界的划分体现的是分析员对软件系统结构的处理方式。 而数据、 输出边界划分与软件结构的处理无关， 仅表示系统与外部数据的交流。 数据输入和输出边界通常是自动化边界的子图（子集）。

  4. 执行一级分解

    所谓一级分解， 是指导出软件逻辑结构的最上两层关系， 顶层为系统主控模块（MC）、第二层根据自动边界划分， 分为输入模块（MI）、输出模块（MO）和控制模块（MT）。

  5. 执行二级分解

  6. 采用启发式规则， 精细化所得到的初步软件结构。

TIP

在实际绘图中， 为了明确说明各模块间的调用关系， 形成接口设计说明书的部分内容， 有必要为每隔模块编写一个简要说明。

通过上述六个步骤得到软件结构图。 这样设计人员可以从整体上把握软件结构， 定义模块间接口， 并为单元测试、 集成测试提供依据。

# 事务分析法

事务分析法与变换分析法设计过程基本相同。

  1. 复审并精华数据流图

  2. 确定数据流图特征， 判断是事务流还是变换流。

    TIP

    事务流的特征是在数据的输入、处理和输出过程中， 处理部分有明显的控制或判断中心， 后续的数据流中有较多活动路径。

  3. 设定自动化边界， 分理出事务中心和事务路径。

  4. 执行一级分解

    事务分析法整体结构是两层结构。 顶层是系统主控模块， 第二层是输入模块和事务调度模块。

  5. 执行二级分解

  6. 采用启发式规则， 精化所得到的初步软件结构。

# 混合分析法

在大规模和复杂系统的数据流图中， 常常需要将变换分析法和事务分析法结合使用， 共同构建软件系统结构图。

混合分析法和变换分析法过程相同， 只是第二步骤时， 需确定数据流图中变换的自动化边界或确定事务中心。

# 结构化详细设计的工具

结构化设计的详细设计阶段，主要完成系统各模块功能的过程描述。 详细设计提供了图形、表格和语言等三类不同工具。

工具 说明 优点 缺点
程序流程图 易懂、易学、易画 1. 控制流（有向箭头）无法约束其转向，造成设计的随意性，并可能导致非结构化的过程设计。
2. 难以表达数据结构， 比如数组就难以表达
盒图 NS 图 1. 控制域明晰，能直接分析出分支、循环的控制域范围。
2. 无控制流，避免了控制流程的随意性。
3. 方块嵌套有层级性
问题分析图 PAD 图 1. 层次清晰，作用域明显
2. 自顶向下， 自左向右易于理解
3. 无控制流<br//>4. 可以表达复杂数据结构
判定树 多叉树，每棵子树就是当前树节点判断的分支路径 1. 对复合条件的选择中， 难以确定以何种顺序作为符合条件的判断顺序
2. 难以表达结构化设计的控制结构
3. 难以将判断树直接转换为程序设计语言
过程描述语言 PDL或者PL(伪码语言)

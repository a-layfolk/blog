# 软件需求工程

用户不会把需求表述完整，所以需要软件需求工程 。现代软件需求工程包括： 问题定义、 可行性分析、 需求分析等阶段。

# 可行性分析

可行性分析的目的， 不是确定问题如何去解， 而是确定问题是否值得去解。

对项目的可行性分析与研究主要包括以下几个方面：

  • 经济可行性： 回答 “系统的经济效益能超过开发成本吗？” 这一问题， 可以使用 代码行技术 和 功能点技术 来估算成本。
  • 技术可行性： 回答 “现有技术能实现这个系统吗？” 这一问题。
  • 操作可行性： 回答 “系统的操作方式符合用户操作流程和方式吗？” 这一问题。
  • 法律可行性： 回答 “系统的开发过程和使用符合当前法律吗？” 这一问题。

在可行性研究阶段， 对系统需求了解的同时， 还需要用概要的形式对现有系统与将来系统的物理模型进行表示。 这就需要用到系统流程图 (System Flowchart) 。

系统流程图表达的是数据在各部件之间的流动情形， 不涉及数据的具体加工。

# 需求分析

需求分析的任务是：

  • 确定系统将要实现的各项要求。 其中最主要、最基本的是功能需求。
  • 数据分析。 将数据从物理实体映射为逻辑实体。
  • 定义逻辑模型：
    • 结构化方法采用数据流图、 数据字典等图形建模工具。
    • 面向对象方法采用用例图、 类-对象等建模工具
  • 适应需求变更

TIP

需求分析的原则个人觉得不重要， 就没记。

需求分析的内容有：

  • 功能需求： 描述系统应该做什么， 有时需要描述系统不应该做什么。 功能描述应该满足完整性和一致性。

    • 完整性： 覆盖用户提出的服务
    • 一致性： 需求不能有矛盾之处

  • 性能需求： 描述系统应该满足在时间和空间上的约束。 比如响应时间、 主存约束等等。

  • 领域需求： 与软件的具体应用范围有关。 比如学号要符合学校的管理规范。

  • 其他需求： 比如法律需求、 道德需求等。

# 需求工程的过程

需求工程在进行可行性研究后， 通过需求获取、需求分析与建模、需求评审的迭代过程， 不断修改得到 《需求规格说明》。

在整个需求工程过程中伴随着需求管理, 它有两个难点:

  • 需求确认困难
  • 需求不断变更

下面分别介绍需求过程的几个阶段：

# 需求获取

需求获取的技术有以下几种：

  • 个别会谈与小组会议
  • 问卷调查
  • 面向用例的场景分析
  • 快速原型技术， 又分为抛弃原型法和演化原型法

# 结构化需求分析与建模

结构化需求分析的核心是数据， 定义在数据字典里。 数据包括了在分析、 设计和实现中涉及的概念、 术语、 属性等所有内容。

围绕数据字典， 完成功能模型、 数据模型和行为模型的结构化建模过程

# 数据字典

数据字典是当前系统的软件词典， 包括各种概念解释和数据描述信息， 目的是让与系统有关的所有人员对信息有共同的、一致的理解。

数据字典要求科学、使用、完成、一致。 再详细的内容可以看书 54 页。

数据字典的定义形式有三种：

  • 词条描述， 分为数据流词条， 数据元素词条， 加工词条和存储文件词条等
  • 定义式：使用巴科斯-诺尔范式来准确、清晰、无二义地定义数据
  • 数据表： 以二维表的形式来表示数据信息， 可以与 E-R 图的数据对象和属性相对应

# 面向数据流的功能建模——数据流图

数据流图（Data Flowing Diagram, DFD）描述从数据输入、数据转换到数据输出的全过程。可以对 DFD 分层 。具体可以看书上第 49 页。

# 面向数据的数据建模——实体关系模型

TIP

结构化数据建模中： 数据对象是属性的集合。

面向对象建模中： 数据对象是属性和行为的集合， 更体现了对象是动静相结合的有机体。

# 面向状态转换的行为建模——状态转换图

状态转换图（Status Transition Diagram, STD）通过描述系统状态及引起状态转换的事件来表示系统状态。

STD 主要由状态、 转换和事件的图形符号组成。

STD 定义了 3 各标准事件， 它们都没有参数：

  • entry 事件
  • exit 事件
  • do 事件

描述状态转换图加工部分的流程或者算法， 可以用 过程描述语言（PDL） 、判定树和判定表。

TIP

过程描述语言（Procedural Description Language, PDL）也称伪码语言（Pseudo-code Language, PL）。即伪代码

# 需求评审

在需求工程完成之前， 必须编写 《软件需求规格说明》和 《数据需求规格说明》， 形成初步的用户手册。

《软件需求规格说明》（Software Requirement Specification, SRS）描述用户的功能需求、性能需求、领域需求等信息。

《数据需求规格说明》（Data Requirement Description, DRD）描述在软件生命周期内所处理的数据， 包括数据采集、存储，数据结构等信息。

需求审核时， 要求 SRS/DFD 具有如下特性：

  • 正确性
  • 完整性
  • 一致性
  • 可行性
  • 可理解性
  • 可验证性
  • 可修改性
  • 可跟踪性

需求变更时， 主要包括以下四个阶段：

  • 确定需求基线
  • 需求变更影响分析
  • 需求变更维护记录
  • 需求变更的稳定、可控性和延续性

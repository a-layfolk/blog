# 版本控制简史

版本控制记录了文件的内容变化，以便将来查阅。

版本控制的发展历程如下：

  • 本地版本控制 一般采用某种简单的数据库记录文件的历次更新差异
  • 集中化的版本控制系统 方便不同系统的开发者协同工作。需要一个单一的集中管理服务器， 保存所有文件的修订版本。 缺点是集中管理服务器宕机时无法协同工作。
  • 分布式版本控制系统 客户端提出最新版本的文件快照的同时， 把代码仓库完整地克隆下来（包括完成的历史记录）， 这样任何一处服务器故障都能够通过本地仓库恢复

# Git 是什么

  • Git 直接记录快照， 而非差异比较（这里我暂时不是很清楚）

  • Git 几乎所有操作都是本地执行

  • Git 保证完整性（不可能在 Git 不知情时更改任何文件内容或目录内容）

  • Git 一般只添加数据（很难使用 Git 从数据库中删除数据， Git 几乎不会执行可能导致文件不可恢复的操作）

  • Git 有三种状态

    • 已修改（modified）， 表示修改了文件， 但还没保存到数据库
    • 已暂存（staged）， 表示对一个已修改文件当前版本做了标记， 使之包含在下次提交的快照中
    • 已提交（committed）， 表示数据已经安全地保存在本地数据库中

    于是 Git 项目拥有三个工作阶段： 工作区、 暂存区以及 Git 目录：

  • Git 的工作流程

    1. 在工作区中修改文件
    2. 将你想要下次提交的更改选择性地暂存， 这样只会将更改的部分添加到暂存区
    3. 提高更新， 找到暂存区的文件， 将快照永久存储到 Git 目录

# 命令行起步

$ git config --list --show-origin # 查看所有的配置以及它们所在的文件

# 设置用户名和邮件地址， 它们会写入到你的每一次提交中， 不可更改
$ git config --global user.name a-layfolk
$ git config --global user.email 1808150553@qq.com

$ git config --list # 列出所有 Git 当时能找到的配置

$ git config <key> # 检查 Git 的某一项配置， 例如 git config user.name

# 三种等价的方法获取 Git 命令的综合手册（manpage）:
$ git help <verb> # 比如想获取 git config 的命令手册， 执行 git help config
$ git <verb> --help
$ man git-<verb>

# 获得可用选项的快速参考
$ git add -h
